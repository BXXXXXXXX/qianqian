# Documentation Rule

更新时间：2026-07-06
负责人：Product
版本：v0.1

---

## 为什么存在？

KTOS 不是普通 App。它同时包含产品设计、AI Prompt、数据结构、幼教知识、UI 设计、开发接口和用户研究。如果全部塞进一份 PRD，项目越往后越难维护。

这个规范的目标是让所有文档可检索、可协作、可被 AI 引用，并且能在团队扩大后保持同一套语言。

---

## 文档原则

- 所有正式文档使用 Markdown。
- 每个功能只允许有一份 canonical document，统一放在 `Features/`。
- 重要系统放在专门目录，不和功能文档重复维护。
- 重要产品选择必须写入 `Decisions/`。
- Prompt、Schema、UI、API 必须写版本。
- 文档可以不完美，但必须留下更新时间、负责人、版本和待优化项。

## 状态定义

- `Draft`：正在形成。
- `Review`：等待团队确认。
- `Active`：当前有效。
- `Deprecated`：不再使用，但保留历史。

---

## 命名规则

- 目录使用编号和英文名，例如 `02_EventEngine/`。
- 功能文档使用英文名，避免多人系统里的路径编码问题，例如 `Growth Record.md`。
- 决策日志使用递增编号，例如 `Decision 001 - Two Main Tabs.md`。
- 版本写在文档头部，不依赖文件名表达全部状态。
